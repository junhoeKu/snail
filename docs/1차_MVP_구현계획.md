# 1차 MVP 구현 계획 — Snail (달팽이 키우기)

> 작성일: 2026-07-10
> 목표 기간: **2~3주**
> 검증 가설: **"사용자가 매일 접속해서 달팽이를 돌보고 싶은가?"**

---

## 1. 개요

### 1.1 프로젝트 정의

달팽이 한 마리를 부화시켜 먹이를 주고 성장시키는 **데일리 케어 게임**.
방치형이 아니라 "하루에 여러 번 들여다볼 이유"를 현실 시간 시스템(배고픔 증가)으로 만든다.

### 1.2 MVP가 검증할 것

핵심 루프 하나가 재미있는가:

```
알 획득 → 부화 → 먹이 주기 → 성장 → 코인 획득 → 먹이 구매 → 더 성장
```

이 루프만 완성한다. 수십 종의 달팽이, 장식, 미니게임은 만들지 않는다.

### 1.3 문서/폴더 규칙

- 모든 기획/설계 문서는 `docs/`에 모은다. `README.md`만 관례에 따라 루트에 둔다.
- 이 문서가 MVP 범위의 단일 기준(spec)이다. 범위 논쟁이 생기면 이 문서를 먼저 갱신한 뒤 구현한다.

## 2. 범위

### 2.1 포함 (MVP 기능 7개)

| # | 기능 | 내용 |
|---|------|------|
| 1 | 달팽이 1종 | 이름 짓기, 레벨, 경험치, 배고픔, 행복도, 성장 단계(알→아기→어린→성체) |
| 2 | 먹이 시스템 | 먹이는 **상추 1종**만. 먹이 주기 → 배고픔 감소 + 경험치 증가 → 레벨업 |
| 3 | 성장 | 경험치 기반 레벨업. Lv5 껍질 커짐, Lv10 색상 변화 |
| 4 | 코인 | 먹이 주기/산책/접속 보상으로 획득. 사용처는 **먹이 구매**뿐 |
| 5 | 시간 시스템 | 현실 시간 기반. 1시간마다 배고픔 +5, 행복도 -5 (미접속 시간도 경과분 정산) |
| 6 | 꾸미기(최소) | 배경 1종 교체 정도. 수십 개 만들지 않는다 |
| 7 | 저장 | LocalStorage 저장. 앱 종료 후 재실행 시 그대로 이어짐 (**필수**) |

### 2.2 기술 제약 (스택 고정)

- **Vanilla HTML/CSS/JavaScript SPA + LocalStorage**로만 진행한다.
- React/Vue/Next/Firebase/Supabase/React Native/게임엔진(Phaser 등)을 붙이지 않는다.
- 빌드 도구/번들러/트랜스파일 없이 정적 파일을 브라우저가 직접 로드한다.
- 외부 라이브러리는 CDN 로드(Font Awesome)만 허용. 새 의존성을 임의로 추가하지 않는다.
- 달팽이 그래픽은 이모지/CSS로 표현한다. 스프라이트·캔버스 애니메이션은 MVP 범위 밖.

### 2.3 제외 (MVP에서 하지 말 것)

- ❌ 달팽이 30종 / 교배 / 희귀 변이
- ❌ 멀티플레이 / 길드 / PvP / 친구 방문
- ❌ 수집 도감 100종 / 업적
- ❌ 시즌 이벤트
- ❌ 광고 보상 / 인앱결제 등 수익화
- ❌ 미니게임
- ❌ 푸시 알림 / 계정 / 서버 저장

전부 재미가 검증된 뒤의 일이다 (§10 로드맵).

## 3. 핵심 루프 & 게임 규칙 (수치)

> 수치는 전부 `GAME.CONFIG` 상수로 모아 한 곳에서 튜닝한다.

### 3.1 스탯 정의

| 스탯 | 범위 | 의미 |
|------|------|------|
| `hunger` | 0~100 | 높을수록 배고픔. 100이면 굶주림 상태 |
| `happiness` | 0~100 | 높을수록 행복 |
| `exp` / `level` | level ≥ 1 | 레벨업 필요 경험치 = `level × 20` |

### 3.2 시간 감쇠 (재접속 유도)

- **1시간 경과마다**: `hunger +5`, `happiness -5`
- 미접속 시간도 `last_seen` 기준으로 경과분을 일괄 정산한다.
- 상한/하한에서 멈춘다 (hunger 최대 100, happiness 최소 0). 죽음/가출 페널티는 MVP에 없다.

### 3.3 행동 규칙

| 행동 | 조건 | 효과 |
|------|------|------|
| 먹이 주기 | 상추 보유, hunger > 0 | `hunger -30`, `exp +10`, `happiness +5`, `코인 +2` |
| 산책 | 쿨다운 4시간 | `happiness +10`, `코인 +10` |
| 접속 보상 | 하루 1회 (날짜 키 `YYYY-MM-DD` 기준) | `코인 +20` |
| 상추 구매 | 코인 ≥ 10 | `코인 -10`, `상추 +1` |

- 경제 균형: 먹이 주기 자체는 코인 순손실(구매 -10, 보상 +2)이고 산책/접속 보상이 주 수입원 → 하루 여러 번 접속할 동기가 된다.
- 시작 자원: **코인 30, 상추 3개**

### 3.4 성장 단계

| 단계 | 조건 | 표현 |
|------|------|------|
| 🥚 알 (`egg`) | 생성 직후 | 이름 짓기 → 부화 |
| 🐌 아기 (`baby`) | Lv 1~4 | 기본 모습 |
| 🐌 어린 (`junior`) | Lv 5~9 | 껍질 커짐 |
| 🐌 성체 (`adult`) | Lv 10+ | 색상 변화 |

- 레벨업/단계 변화 시 축하 연출(토스트 + 간단한 CSS 애니메이션)을 보여준다. 사용자는 성장하는 모습만 봐도 만족감을 느낀다.

## 4. 데이터 모델 (LocalStorage)

| 키 | 모듈 | 내용 |
|------|------|------|
| `sn_player` | `DB.Player` | `coins`, `food`(상추 수), `last_seen`(ISO), `last_daily_reward`(YYYY-MM-DD), `last_walk`(ISO), `background` |
| `sn_snail` | `DB.Snail` | `name`, `level`, `exp`, `hunger`, `happiness`, `stage`(egg/baby/junior/adult), `color`, `created_at`(ISO) |

- 각 레코드에 `schema_version` 필드를 넣어 이후 마이그레이션(백엔드 DB 이전 포함)에 대비한다.
- 날짜 키는 `YYYY-MM-DD`(로컬 시간), 타임스탬프는 ISO 문자열로 통일한다.
- 파싱 실패/용량 초과 시 기본값 반환 + 콘솔 경고. 사용자 데이터 전체 삭제로 대응하지 않는다.

## 5. 도메인 로직 (`game.js` = `GAME`, 순수 함수)

DOM/DB에 접근하지 않는다. 입력을 받아 새 상태와 이벤트 목록만 반환한다.

| 함수 | 역할 |
|------|------|
| `GAME.CONFIG` | §3의 모든 수치 상수 |
| `GAME.STAGES` | 성장 단계 정의 (조건/표시 정보) |
| `GAME.hatch(snail, name)` | 알 → 아기 부화 |
| `GAME.feed(snail, player)` | 먹이 주기. `{ snail, player, events }` 반환 |
| `GAME.walk(snail, player, nowISO)` | 산책 (쿨다운 검사 포함) |
| `GAME.claimDaily(player, todayKey)` | 일일 접속 보상 |
| `GAME.applyTimeDecay(snail, lastSeenISO, nowISO)` | 경과 시간만큼 배고픔/행복도 정산 |
| `GAME.gainExp(snail, amount)` | 경험치 → 레벨업 → 단계 변화까지 처리, `events`에 기록 |
| `GAME.expToNext(level)` | 다음 레벨 필요 경험치 |
| `GAME.buyFood(player)` | 상추 구매 |

- 모든 행동 함수는 `{ snail, player, events }` 형태를 반환한다. `events` 배열(예: `["fed", "levelup", "stage_up"]`)을 화면 모듈이 받아 토스트/연출을 처리한다.
- 실패 시(코인 부족, 쿨다운 등) 상태를 바꾸지 않고 `events`에 실패 사유(예: `"not_enough_coins"`)를 담아 반환한다.

## 6. 화면 구성 (3개)

```
[홈]                        [상점]               [성장 연출]
🐌 달팽이 (단계별 모습)       🥬 상추  10코인       Lv UP!
Lv.5  😊 행복 80%            [구매]               껍질이 커졌습니다!
🍀 배고픔 20%                                     (홈 위 오버레이)
[먹이주기] [산책] [상점]
```

- **홈**: 달팽이 상태 표시 + 먹이주기/산책 버튼 + 상점 진입. 알 상태면 이름 짓기/부화 UI.
- **상점**: 상추 구매 (+ 배경 교체 1종, 선택 사항).
- **성장 연출**: 별도 페이지가 아닌 홈 위 모달/오버레이로 처리.
- 첫 실행(데이터 없음) 시 알 지급 → 이름 짓기 온보딩으로 시작한다.

## 7. 파일 구조 (목표)

```
snail/
├── index.html          ← SPA 진입점. 모든 CSS/JS 로드 + 화면 컨테이너
├── css/
│   ├── theme.css       ← 색상 변수/배경
│   ├── components.css  ← 버튼/카드/모달/토스트/스탯 바
│   ├── home.css        ← 홈(달팽이) 화면
│   └── shop.css        ← 상점 화면
├── js/
│   ├── db.js           ← ★ LocalStorage 데이터 계층 (Player/Snail)
│   ├── game.js         ← ★ 핵심 도메인 로직 (순수 함수)
│   ├── home.js         ← 홈 화면 렌더링 + 행동 버튼
│   ├── shop.js         ← 상점 화면
│   ├── toast.js        ← 토스트/모달/성장 연출
│   └── app.js          ← 앱 컨트롤러 (부팅/시간 정산/라우팅)
├── docs/
│   └── 1차_MVP_구현계획.md
└── README.md
```

- 1 JS 파일 = 1 IIFE = 1 전역 네임스페이스: `DB`, `GAME`, `HomeModule`, `ShopModule`, `Toast`, `App`

## 8. 구현 단계 (브랜치 단위)

| 단계 | 브랜치 | 내용 | 완료 기준 |
|------|--------|------|-----------|
| 1 | `feat/skeleton` | index.html + CSS 뼈대 + `App` 부팅/화면 전환 | 빈 홈/상점 화면 전환 동작 |
| 2 | `feat/db` | `DB` 계층 (Player/Snail CRUD, 기본값, 방어 코드) | 콘솔에서 저장/조회, 새로고침 후 유지 확인 |
| 3 | `feat/game-core` | `GAME` 순수 함수 전체 (§5) | 콘솔에서 핵심 루프 전체 시뮬레이션 가능 |
| 4 | `feat/home-ui` | 홈 화면 + 부화 온보딩 + 먹이/산책 버튼 | 핵심 루프가 화면에서 동작 |
| 5 | `feat/shop` | 코인 표시 + 상점 + 상추 구매 | 코인 순환(획득→구매→소비) 동작 |
| 6 | `feat/growth-fx` | 레벨업/단계 변화 연출 + 시간 감쇠 정산 | 재접속 시 배고픔 증가 반영 |
| 7 | `chore/polish` | 모바일 390px 대응, QA, README | §9 배포 기준 통과 |

```bash

```

```js
// 브라우저 DevTools Console 검증
DB.Player.get();
DB.Snail.get();
GAME.feed(DB.Snail.get(), DB.Player.get());
```

- 콘솔 에러 0개
- 새로고침 후 데이터 유지
- LocalStorage 삭제 후 알/이름짓기 온보딩 정상 시작
- 미접속 후 재접속 시 배고픔/행복도 정산 정확
